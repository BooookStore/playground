# Ktor Matching Service

## これはなに？

個人勉強を目的とし「マッチング最適化」問題を題材にしたアプリケーションです。単純すぎると面白くないので、以下のような仮想の要求を想定し一定の複雑度を加えました。

ある会社の営業組織では商談の模擬練習を行っています。模擬練習は社員同士が商談をする側とされる側になりきって、二人一組のロールプレイで行われています。このロールプレイの組み合わせを作成するのがこのアプリーケーションの目的です。

社員の組み合わせには条件があり、条件は会社の判断に応じて追加・削除・変更されることが高い頻度で発生するものとします。以下は条件の一部です。

- 過去の組み合わせをできるだけ除く。慣れが発生してしまって練習としての質が落ちてしまうからだ。
- 営業としての経験が浅い人は、高い人とできるだけ組み合わさるようにする。熟練者から良いフィードバックを得ることができるからだ。
- できるだけ同じ支部の社員同士の組にする。物理的な移動コスト（時間や費用）を避けるためだ。
- 出張により模擬練習に参加することが難しい社員は対象から外す。

## どんな設計にしたの？（もしくはする予定？）

### 1. 組み合わせ条件

この題材で最も変化しやすい箇所は「組み合わせの条件」だと仮定しました。現実のアプリケーションにおいても、「ある条件に基づいて」の部分はビジネスの戦略に影響を受けやすいからです。そのため、この題材においても条件の追加・削除・変更がより高い確率で発生するものと捉えてみます。

この前提に立ってみると、組み合わせ条件の変更は他の箇所と比べると変更する頻度が高い **ホットスポット** と言えそうです。ホットスポットは、早期に抽象化を行うべき価値が高い領域であるため、優先して適切な設計を施すようにしてみました。
